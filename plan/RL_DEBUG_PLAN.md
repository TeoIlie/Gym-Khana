# RL Training Debug Plan

Currently the drift controller is not training as expected. The following are the top candidates for fixing this issue:

| Status | Issue | Fix |
|--------|-------|-----|
|[ ]| Simple controller not working - PPO trained for 2 mil steps on `train_ppo_example.py` did not converge | Begin with a simple controller, like the original `ppo_example.py` - train this and ensure it works, then iterate: <ol><li> First use single dummy env, then parallelize</li><li> Go from simple map like `IMS` to challenging map like `Spielberg` </li><li> Upgrade from `st` to `std` model to validate the model is working </li><li>Begin with a minimal observation subset and slowly increase</li></ol>|
|[ ]| Incorrect observation normalization | <ol><li> After action normalization, normalizing actions like `steering` and `acceleration` commands may require new boundaries -> `[-1,1]` instead of `[min_steer, max_steer]` </li><li> Some observations I only guessed their bounds. These need to be validated with command `record_obs_min_max`. For this to work, it must be adjusted to work for a parallelized `SubprocVecEnv`, maybe by creating a custom wrapper that can aggregate the min/max dictionary values across all parallel environments. </li><li>  Use a gym wrapper like `VecNormalize` to auto-generate bounds and compare </li></ol>|
|[ ]| Common RL issues | <ol><li> Insufficient steps - add `EvalCallback` or use SB3 evaluation helper function <https://stable-baselines3.readthedocs.io/en/master/common/evaluation.html#eval> to evaluate periodically</li><li>Consider early stoppage condition to train longer until convergence </li><li> Hyperparameter choice - implement hyperparameter optimization strategies like sweeps - use **Optuna** <https://optuna.org/></li><li> Consider reward normalization</li><li>Speed training with **SBX** <https://github.com/araffin/sbx></li></ol> |
|[ ]| Reset function | Look into how other papers implemented their reset - resetting on track boundary violation may not allow enough learning |
|[ ]| Throttle | To simplify the search at first, consider not allowing negative throttle, so the car can only go forward without braking (accelerating or coasting) |
|[ ]| Reward function | <ol><li>**My idea:** reward progress along centerline, and penalize out of bounds proportional to how far out of bounds the car is, using track width and Frenet lateral deviation. Add tunable gains for both centerline progress and out of bounds penalization, possible as gym env config options</li><li> **Scaramuzza Gran Turismo reward function:** Instead of the current reward function, first set `wall_deflection` flag to `True` so the car cannot exceed track boundaries (perhaps widen boundaries too if needed). Then, penalize wall collisions in proportion to the speed of impact</li><lo> |
|[ ]| The paper is leaving out some important implementation details, and replication it is not enough | What then...? How do other papers solve the problem? |
|[ ]| Alternate DRL algos | Try SAC instead - some racing papers use it instead of PPO.
|[ ]| Additional unit tests | Test RL config, callbacks, custom activation function, check env with SB3 `check_env` method |
